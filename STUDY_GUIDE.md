# KGPBoard — Study / Revision Guide

A full-stack **campus notices & events** platform. Use this to revise before
your interview. Structure: what it is → architecture → backend → frontend →
deployment → key decisions → likely Q&A → glossary.

---

## 1. The 30-second pitch

> "KGPBoard is a full-stack app where students browse campus notices and events,
> and admins post them. The backend is a REST API built with **FastAPI** and
> **PostgreSQL** (fully async), organised in clean layers. The frontend is a
> responsive **React + TypeScript** dashboard. It has JWT auth with roles,
> filtering, pagination, image uploads, and it's deployed live (React on Vercel,
> API + Postgres on Render)."

---

## 2. Architecture overview

```
Browser (React app on Vercel)
      │  HTTPS + JSON
      ▼
FastAPI backend on Render  ──►  PostgreSQL on Render
```

**The backend is layered** — each layer has ONE job, and only talks to the
layer directly below it:

```
HTTP request
   ▼  api/routes/    → HTTP: parse the request, validate it, return a response
   ▼  services/      → business rules, transactions, "what is allowed"
   ▼  repositories/  → the ONLY place that runs SQL / touches the database
   ▼  models/        → the tables + database constraints
PostgreSQL
```

**Why layers?** Separation of concerns. Each layer is testable and swappable.
If I change the database, only `repositories/` changes. If I add a rule, it goes
in `services/`. Routes stay thin. This is the single most important thing to be
able to explain.

## 3. Backend — deep dive

### 3.1 Tech stack (and why)
- **FastAPI** — modern async Python web framework; auto-generates OpenAPI docs.
- **PostgreSQL** — relational DB; strong constraints (unique, foreign keys, checks).
- **SQLAlchemy 2.0 (async)** — the ORM: write Python classes, it generates SQL.
- **asyncpg** — the async PostgreSQL driver SQLAlchemy talks through.
- **Alembic** — database migrations (versioned schema changes).
- **Pydantic v2** — validates request/response data from type hints.
- **PyJWT + Argon2 (pwdlib)** — tokens + password hashing.
- **pytest** — testing.

### 3.2 The layers, one by one

**Models** (`app/models/`) — SQLAlchemy classes = database tables.
- `User` (id, name, email UNIQUE, role enum, password_hash, createdAt)
- `Notice` (…, category enum, postedBy → User FK, imageUrl?)
- `Event` (…, category enum, venue, startTime, endTime, organizer, imageUrl?,
  registrationUrl?, targetYear?)
- Constraints live here: `UNIQUE(email)`, foreign keys, and a **CHECK** that
  `start_time < end_time` — the DB physically refuses a bad event.
- IDs are **UUIDs** (generated by Postgres `gen_random_uuid()`) — non-guessable,
  better than auto-increment integers for a public API.

**Repositories** (`app/repositories/`) — the Data Access layer (DAO). The ONLY
code that writes SQL. Functions like `create`, `get_by_id`, `list_`. They
`flush()` (send SQL, surface DB errors) but do **not** `commit()` — the caller
owns the transaction.

**Services** (`app/services/`) — business logic. Enforce rules (email unique,
startTime < endTime), own the **transaction** (`commit`/`rollback`), and raise
**domain errors** (`EmailAlreadyExistsError`, `InvalidTimeRangeError`) that know
nothing about HTTP.

**Routes** (`app/api/routes/`) — thin HTTP endpoints. Parse the request into a
Pydantic schema, call a service, return the result. An error-handler layer maps
each domain error to an HTTP status.

### 3.3 Validation — done twice, on purpose
1. **At the edge (Pydantic schemas):** required fields, email format (`EmailStr`),
   types, category is one of the fixed enum values. Bad input → `400` before any
   logic runs.
2. **At the database (constraints):** UNIQUE email, the `start < end` CHECK. This
   is the *last line of defense* — even a bug in app code can't store bad data.

Errors always return a JSON `detail` message (never just a status code), and
validation errors list each bad field.

### 3.4 Async — why and how
- "Async" lets one process handle many requests concurrently: while one request
  waits on the database, the server serves others. Great for I/O-bound APIs.
- Every DB call is `await`ed; the engine is `create_async_engine`, sessions are
  `AsyncSession`, the driver is `asyncpg`.

### 3.5 Authentication & authorization (the big one)
- **Authentication** = *who are you?* **Authorization** = *what may you do?*
- **Passwords are hashed** with **Argon2** (one-way). We never store the raw
  password; on login we re-hash the input and compare.
- **Login** returns a **JWT** (JSON Web Token) — a *signed* (not encrypted) token
  carrying the user id + role + expiry. The client sends it as
  `Authorization: Bearer <token>`. We trust it because the signature proves we
  issued it (HS256, signed with a secret).
- **FastAPI dependencies** enforce access:
  - `get_current_user` → decodes the token, loads the user, else `401`.
  - `require_admin` → builds on that; non-admin → `403`.
- **Rule:** only admins create notices/events; reads are public. Public sign-up
  always makes a **student** (so nobody can self-promote to admin). Admins are
  seeded / created by an existing admin.
- Login errors are identical for "unknown email" vs "wrong password" → prevents
  **user enumeration**.

### 3.6 Filtering & pagination
- List endpoints accept `category`, `from`/`to` (date range), `q` (keyword),
  `limit`, `offset`.
- They return an **envelope**: `{ items, total, limit, offset }` so the client
  knows how many exist.
- Implementation: build a list of WHERE conditions from the non-empty filters,
  apply to both a `COUNT(*)` query and the paged `SELECT`.

### 3.7 Categories as enums
- `category` is a **fixed Postgres enum** per type (notices vs events have
  different sets). Invalid category → `400` listing the valid options. A `/meta`
  endpoint exposes the option lists so the frontend dropdowns stay in sync with
  the backend — single source of truth.

### 3.8 Image uploads (storage abstraction)
- `POST /uploads` (admin) validates type (JPEG/PNG/WebP) + size (≤5 MB), stores
  under a **random filename** (prevents path-traversal), returns a URL.
- A **Storage abstraction** (`LocalStorage` today) means swapping to S3/Cloud
  storage later is one class — nothing else changes. Files served as static.

### 3.9 Rate limiting
- Write endpoints capped at `5/minute` (SlowAPI) → `429 Too Many Requests`.
  Keyed per-user when logged in, else per-IP.

### 3.10 Migrations (Alembic)
- The DB schema is versioned. Change a model → generate a migration → apply with
  `alembic upgrade head`. Every environment gets the same schema reproducibly.

### 3.11 Testing
- **39 pytest tests** against a throwaway `kgpboard_test` database (created/
  dropped automatically) so real data is never touched.
- Cover validation, auth (401/403/201), filtering, pagination, role changes.
- Key trick: **dependency override** — swap the app's DB session for a test one.

### 3.12 User management (admin)
- `GET /users` (admin only) — search users by email, paginated.
- `PATCH /users/{id}/role` (admin) — change a user's role, **blocked for your own
  account** (so the last admin can't lock everyone out).

## 4. Frontend — deep dive

### 4.1 Tech stack (and why)
- **Vite + React + TypeScript** — fast dev, typed components.
- **React Router** — client-side routing (deep-linkable pages).
- **TanStack Query** — data fetching with automatic loading/error/caching.
- **Plain CSS with variables** — theming + dark mode via CSS custom properties.

### 4.2 Architecture (portable layer)
```
src/
├── types.ts       │ shared API types  ┐ NO DOM code — a future React Native
├── api/client.ts  │ typed fetch       │ mobile app could reuse this layer
├── hooks/         │ TanStack Query     ┘ verbatim
├── components/    │ header, cards, feeds, states, footer
├── pages/         │ dashboard, detail, login, signup, create, admin users
├── auth/          │ AuthProvider (token in localStorage)
└── theme/         │ dark mode
```
This mirrors the backend's discipline: keep data/logic separate from UI.

### 4.3 Key concepts
- **Loading/error states**: TanStack Query gives `isPending / isError / data`,
  which map directly to skeletons / error+retry / the list.
- **Pagination**: `useInfiniteQuery` + a "Load more" button.
- **Search + category filter** work together, stored in the **URL** (`?q=&category=`)
  so they're shareable and survive refresh.
- **Detail pages** (`/events/:id`) fetch a single item by id → deep-linkable.
- **Dark mode**: CSS variables redefined under `[data-theme="dark"]`; preference
  saved in `localStorage`; an inline script sets it before paint (no flash).
- **Auth**: `AuthProvider` stores the JWT, attaches it to requests, restores the
  session on reload via `/auth/me`.
- **Admin-only UI**: the `＋ New` and `Users` buttons and their pages are shown
  and route-guarded only for `role === "admin"`.

## 5. Deployment
- **Frontend → Vercel** (static build of `web/`), env `VITE_API_URL`.
- **Backend + Postgres → Render** (from `render.yaml` blueprint; Docker image;
  migrations run on boot).
- **CORS**: the backend must list the frontend's origin (`CORS_ORIGINS`) or the
  browser blocks cross-origin calls.
- Config comes from **environment variables** (`DATABASE_URL`, `JWT_SECRET`,
  `CORS_ORIGINS`) — no secrets in code. The app auto-rewrites a `postgres://`
  URL to the async `postgresql+asyncpg://` driver.

## 6. Key decisions & trade-offs (be ready to justify these)
- **Layered architecture** — testable, swappable, single-responsibility.
- **UUID ids** — non-guessable, safe for a public API.
- **Validate twice** (schema + DB) — friendly errors *and* guaranteed correctness.
- **Enums for category** — reliable filtering, no typo categories.
- **Domain errors, not HTTP in services** — services reusable outside the web.
- **Storage abstraction** — swap local disk for cloud with one change.
- **Async** — handles concurrent requests efficiently (I/O-bound workload).
- **TanStack Query** — loading/error/caching for free vs hand-rolled `useEffect`.

## 7. Likely interview questions + answers

**Q: Walk me through what happens when a student creates a user.**
A: Request → Pydantic `UserCreate` validates required fields + email format →
`user_service.create_user` checks the email isn't taken, hashes the password,
calls the repository to insert, commits → returns a `UserRead` (never the
password). If the email exists → `EmailAlreadyExistsError` → mapped to `409`.

**Q: Why separate repositories and services?**
A: Repositories know *how* to persist (SQL). Services know *what the operation
means* (rules, transactions, coordinating multiple repos). Keeps SQL out of
business logic; either can change independently.

**Q: How does auth work?**
A: Login verifies the Argon2 password hash and returns a signed JWT with the
user id + role. Protected routes use a dependency that decodes the token and
loads the user (401 if invalid); an admin-gate dependency checks the role (403).

**Q: Why is the password hashed and not encrypted?**
A: Hashing is one-way — even we can't read it. We only ever compare hashes, so a
DB leak doesn't expose passwords. Encryption is reversible; wrong tool here.

**Q: How do you prevent bad data (e.g. an event ending before it starts)?**
A: The service checks `startTime < endTime` for a friendly 400, AND the database
has a CHECK constraint as a guaranteed backstop.

**Q: What's a migration and why use one?**
A: A versioned, reproducible change to the DB schema (Alembic). Everyone applies
the same steps; no manual "ALTER TABLE" drift between environments.

**Q: How does the frontend handle loading and errors?**
A: TanStack Query exposes `isPending / isError / data`; I render skeleton cards,
an error message with a retry button, or the list respectively.

**Q: What's CORS and why did you need it?**
A: Browsers block a page on one origin (Vercel) from calling an API on another
(Render) unless the API says it's allowed. I set `CORS_ORIGINS` to the frontend
URL so the browser permits the requests.

**Q: How would you scale / what would you improve?**
A: Move uploads to object storage (S3/R2), add Redis-backed rate limiting and
caching, add a read replica, cursor-based pagination for huge lists, and CI.

## 8. Glossary (quick hits)
- **REST API** — HTTP endpoints returning JSON; verbs map to actions (GET/POST…).
- **ORM** — Object-Relational Mapper; Python objects ↔ DB rows (SQLAlchemy).
- **Async / await** — non-blocking I/O; serve other requests while waiting on DB.
- **JWT** — signed token proving identity; carries claims (id, role, exp).
- **Hashing** — one-way transform (passwords). **Signing** — proves authenticity.
- **Migration** — versioned schema change.
- **CORS** — browser cross-origin permission mechanism.
- **CRUD** — Create, Read, Update, Delete.
- **Dependency injection** — FastAPI supplies things (DB session, current user)
  to endpoints automatically.
- **Idempotent** — same request repeated has the same effect (e.g. GET).
- **Constraint** — a DB-enforced rule (UNIQUE, FK, CHECK).
